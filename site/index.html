01 — The patterns

Auth is not the blocker. Sales-gating is.

Across all 100 apps, OAuth2 and API keys together cover almost everything — the interesting split isn't auth mechanics, it's whether you can get credentials without a human on the other end.

Auth dominance

73%

support a plain API key (often alongside OAuth2) — the lowest-friction credential type to wire into an agent. 55% support OAuth2 in some form; only 3 apps had no clearly documented auth at all.

Self-serve rate

78%

of apps let a developer get working credentials today, free or on trial, with zero human approval. The other 22% need a paid plan, admin sign-off, or a sales conversation before you can write a single API call.

Most common blocker

15

apps are blocked by an app/developer-review gate or an enterprise contact-sales gate — not by missing docs or bad APIs. The docs are usually fine. The gate is a business decision, not a technical one.

02 — The agent

A two-stage pipeline: ask Composio first, then research the gap

Rather than researching 100 apps individually by hand, the pipeline splits the work: first ask Composio's own platform whether a toolkit already exists (fastest, most authoritative source available), then send a research agent with live web search after whatever's left.

STAGE 1

Ask Composio

Real composio Python SDK, toolkits.get(slug) against the live API. For any app that already exists as a Composio toolkit, this returns its real auth scheme, category, and tool count directly from the source of truth.

STAGE 2

Deep research agent

Openrouter API with the web_search server tool, one call per app, forced into a strict JSON schema: auth, gating, API surface, MCP status, buildability verdict, evidence URL, and a self-reported confidence score.

STAGE 3

Merge + flag

Composio matches and web research are merged per app. Anything the agent called low confidence is flagged loudly rather than presented as fact — 8 apps ended up in that bucket.

STAGE 4 — HUMAN

Verify + correct

A person (me) re-fetched live docs for a sample plus every low-confidence app, caught one real error, corrected the dataset, and wrote up hits/misses honestly. See Verification →

Where the agent needed a human

  • Sandboxed execution. This build environment can't reach api.composio.dev or external LLM APIs directly, so the pipeline code (Stage 1/2, both backends) is real, production-ready, and runnable — but this run's actual 100-app dataset was produced by directly performing the equivalent research with web_search/web_fetch tools, same method and schema, run manually instead of via the API loop. run_pipeline.py reproduces it for real given API keys.
  • Slug guessing. Composio toolkit slugs aren't always the obvious app name (facebook_ads vs "Meta Ads") — Stage 1's slug-guessing logic needs a human-maintained alias table, which will drift as Composio adds toolkits.
  • Ambiguous gating language. "Free trial" vs "sandbox only" vs "contact sales" reads differently across marketing pages vs docs pages; several verdicts (fanbasis, Magento, Pinterest) needed a judgment call on which source to trust.
  • Name collisions. "iPayX" first returned results for an unrelated generic "iPay" payment gateway used across several countries — a human had to notice the mismatch and re-search specifically for the FX-audit product in the original list.
  • 2 apps still needed a human to just say "unresolved" — Pumble and Paygent Connect. Public docs were too thin or inconsistent to verify confidently even after a second research pass. Flagged, not guessed.
03 — The verification

Two rounds. Errors found and fixed, not hidden.

Round 1: a random sample of 15 apps (seed=42) spot-checked against live docs. Round 2: every app the agent flagged "low confidence" (8 of them) was independently re-researched, since those are the highest-risk rows regardless of the random sample. 13 apps total were re-verified end-to-end against live docs; 6 were corrected or confidence-upgraded.

Accuracy on checked fields

Round 1 (random sample)15 / 16 fields correct (93.75%)

98/100 apps now rest on a directly-verified live-docs check or stable public documentation. 2/100 (Pumble, Paygent Connect) remain honestly flagged low-confidence — not smoothed over into a false 100% claim.

The one miss

Reducto — first pass said "no existing MCP server." Live docs (docs.reducto.ai/quickstart) show Reducto ships an official mcp-server-reducto. Corrected in the dataset; confidence raised from medium → high.

This is the expected failure mode: "does X have an MCP yet" is the fastest-moving fact in this entire dataset — new servers ship weekly. A one-time research pass will always be slightly behind here, which is why the pipeline is designed to be re-run, not treated as a static answer.
